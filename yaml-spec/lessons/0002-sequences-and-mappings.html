Sequences & Mappings — building structure

Every config file is a tree of these two collection types. Know them cold.

⏱ ~15 min 🎯 Block/flow styles, nesting, yaml.v3 struct tags, gotchas

The two collection types

YAML has exactly two ways to group values:

  • A sequence is an ordered list of values. Think []T in Go.
  • A mapping is an unordered set of key–value pairs. Think map[K]V or a struct in Go.

Both can be written in block style (one entry per line, indented) or flow style (inline, bracketed). The meaning is identical — style is a readability choice.

Sequences

Block style

Each item is introduced by a dash and a space (- ). The dash is the block sequence entry indicator.

servers:
  - alpha.example.com
  - beta.example.com
  - gamma.example.com

Indentation matters: every - must sit at the same column to be part of the same sequence. The items themselves (here, plain scalars) can be more complex — a mapping, another sequence, anything.

Flow style

Wrap items in square brackets, separated by commas. Useful for short, homogeneous lists:

allowed_envs: [dev, staging, prod]
port_range:   [8080, 8443, 9000]
Style convention

Use block style for sequences of complex items (mappings, multi-line strings). Use flow style only for short, flat lists of scalars — it signals "this is a simple enumeration."

Mappings

Block style

Each entry is key: value, one per line. The space after the colon is mandatory — without it YAML treats the entire token as a scalar.

database:
  host:     localhost
  port:     5432
  name:     myapp
  ssl:      true

Flow style

Wrap entries in curly braces, comma-separated. Good for small, flat inline objects:

tls: {cert: cert.pem, key: key.pem}
The space after the colon

host:localhost (no space) is a single scalar with value "host:localhost", not a mapping entry. The colon is only a mapping indicator when followed by a space (or end of line).

Nesting collections

Sequences and mappings compose freely. A real CLI config typically looks like this:

app:
  name:    my-cli-tool
  version: "1.0.0"

log:
  level:   info
  outputs:
    - stdout
    - /var/log/my-cli-tool.log

plugins:
  - name:    audit
    enabled: true
  - name:    cache
    enabled: false

The plugins key holds a sequence of mappings. Note the indentation: the mapping keys (name, enabled) sit indented relative to their -. YAML allows the keys to start on the same line as the dash or on the next.

Keys on same line as dash

- name: audit
  enabled: true

Keys on next line (explicit)

-
  name: audit
  enabled: true

Both are valid and equivalent. The first is more common; the second is useful when the first key name is long and you want them all aligned.

How yaml.v3 maps these to Go

Sequences → slices

A YAML sequence unmarshals into a Go []T where T is inferred from the first element, or interface{} when the type is mixed or unknown:

// Config Go struct
type Config struct {
  Servers []string  `yaml:"servers"`
  Plugins []Plugin  `yaml:"plugins"`
}

Mappings → structs or maps

A YAML mapping unmarshals into a Go struct (for known schemas) or a map (for dynamic/unknown keys):

// Struct: schema is known at compile time
type Database struct {
  Host string `yaml:"host"`
  Port int    `yaml:"port"`
}

// map: dynamic keys (e.g. user-defined labels)
type Config struct {
  Labels map[string]string `yaml:"labels"`
}
interface{} and map key type

When yaml.v3 unmarshals a mapping into an interface{}, the concrete type is map[string]interface{} — not map[interface{}]interface{} as yaml.v2 used to produce. This is a breaking change between v2 and v3 that trips people up when migrating.

The essential struct tags

yaml:"name" — use this key name instead of the lowercased field name yaml:",omitempty" — skip the field when marshalling if it's the zero value yaml:",inline" — flatten a nested struct into the parent mapping yaml:"-" — never marshal or unmarshal this field

You can combine the name and options: yaml:"host,omitempty".

type Server struct {
  Host     string  `yaml:"host"`
  Port     int     `yaml:"port,omitempty"`  // omit when 0
  internal string  `yaml:"-"`               // never in config
}

type Config struct {
  Server  // inline: Host/Port appear at the top level
  Log    LogConfig `yaml:"log,inline"`
}
Default key name

Without a yaml: tag, yaml.v3 lowercases the field name: field MaxRetries maps to key maxretries (no separator). Use explicit tags for anything other than simple lowercase names.

Gotchas

Duplicate mapping keys

The YAML spec says duplicate keys should not be used — but doesn't forbid them. yaml.v3 accepts them silently, with the last value winning:

port: 5432
port: 3306   # this wins — no error, no warning
Duplicate key danger

A misconfigured file with two port keys won't surface an error. Validate your config schema programmatically (e.g. with KnownFields or a struct validator) rather than relying on the parser to catch it.

Null values in sequences

A bare - with nothing after it is a null entry, not an empty string:

tags:
  - prod
  -          # null — unmarshals to "" for string, 0 for int
  - stable

Map key ordering

YAML mappings are unordered by definition. yaml.v3 does not preserve insertion order when unmarshalling into a map. If order matters (e.g. a list of named steps), use a sequence of mappings instead of a mapping:

# Bad — map iteration order is undefined
steps:
  checkout: 1
  build:    2
  test:     3

# Good — sequence preserves order
steps:
  - checkout
  - build
  - test
