Block Scalars — multi-line strings done right

The two block scalar styles give you precise control over newlines in config values.

⏱ ~15 min 🎯 Literal |, folded >, chomping indicators, indentation

Why block scalars exist

Sometimes a config value is a paragraph of text, a shell snippet, a SQL query, or an embedded template. Cramming it into a single-line double-quoted string works, but it's unreadable and brittle. Block scalars solve this cleanly:

# Terrible — hard to read, easy to break
help_text: "Run this tool with --config pointing\nto your config file.\nSee the docs for details."

# Block literal — exactly what you want
help_text: |
  Run this tool with --config pointing
  to your config file.
  See the docs for details.

There are two block scalar styles. They differ in one thing: what happens to newlines within the value.

Literal style — |

The pipe | indicator means "preserve every newline." What you write is what you get, character for character (modulo the leading indentation, which is stripped).

sql: |
  SELECT id, name
  FROM   users
  WHERE  active = true
  ORDER  BY name;

yaml.v3 delivers this as the Go string:

"SELECT id, name\nFROM   users\nWHERE  active = true\nORDER  BY name;\n"

Note the trailing \n — by default, block scalars end with exactly one newline. This is called clip chomping and is the default.

Folded style — >

The greater-than > indicator means "fold single newlines into spaces." This lets you wrap long text in the config file for readability without embedding literal newlines in the value.

description: >
  This is a long description that wraps
  across multiple lines for readability,
  but will be delivered as one paragraph.

Go receives:

"This is a long description that wraps across multiple lines for readability, but will be delivered as one paragraph.\n"

The folding rule has one exception: blank lines and more-indented lines are preserved. A blank line becomes a literal \n. A line indented further than the block's base becomes a literal newline too (useful for embedded code or lists).

notes: >
  First paragraph, folded into
  one line.

  Second paragraph — blank line above
  preserved as a newline.

    This line is extra-indented,
    so its newlines are kept literal.

Chomping — controlling the trailing newlines

Every block scalar ends with some newlines. The chomping indicator — a character added right after | or > — controls how many trailing newlines survive in the final string:

| clip (default) — one trailing newline |- strip — zero trailing newlines |+ keep — all trailing newlines preserved Same three options work with >, >-, and >+.
# Clip (default): one trailing \n
a: |
  hello

# Strip: no trailing \n — useful when the value feeds a shell command
b: |-
  hello

# Keep: all trailing blank lines become \n
c: |+
  hello
When strip matters

If your Go code passes the config value directly to exec.Command or a template, a trailing newline is usually a bug. Use |- so the value is clean without requiring strings.TrimSpace in Go.

Explicit indentation indicator

Normally YAML infers the indentation level from the first non-empty line. You can override this by appending a digit (1–9) to the block indicator — this sets the number of leading spaces that are stripped:

script: |2
    echo "indented by 2, not 4"

This is rarely needed. The main case is when the first content line is blank (YAML can't infer indentation from a blank line) or when you need to embed content that starts with spaces you want to preserve.

The chomping and indentation indicators can be combined in either order: |2- and |-2 are both valid.

Literal vs folded — decision guide

Use literal | when

  • Line breaks are meaningful: shell scripts, SQL, code snippets
  • Embedded templates where \n controls rendering
  • The exact whitespace matters

Use folded > when

  • The value is prose: help text, descriptions, messages
  • You want to wrap long strings in the config file for readability
  • Line breaks in the config are for the author, not the consumer

yaml.v3 and block scalars

yaml.v3 unmarshals block scalars into Go string (or []byte) with no special handling — the chomping and folding are resolved entirely by the parser. The Go field just receives the final string:

type Config struct {
  Script      string `yaml:"script"`       // gets the block scalar value
  Description string `yaml:"description"`
}

When marshalling, yaml.v3 automatically chooses block literal style for strings that contain newlines, so a round-trip preserves multi-line values correctly.
