Documents & Streams

A YAML file is a stream. A stream can hold multiple documents. Know the boundaries.

⏱ ~12 min 🎯 ---, ..., directives, yaml.v3 Decoder, multi-doc gotchas

The three-level model

YAML defines a strict hierarchy that most people never think about because they only ever see one document per file:

Stream └─ Document 1 └─ Root node (a mapping, sequence, or scalar) └─ Document 2 └─ Root node └─ ...

A stream is the entire byte sequence you hand to a parser. A document is one self-contained YAML value within that stream. Every file you've written so far has been a stream containing exactly one implicit document.

Document markers — --- and ...

--- (three dashes) is the document start marker. It explicitly begins a new document, ending any previous one.

... (three dots) is the document end marker. It explicitly closes the current document without starting a new one.

---                    # start of document 1
name: alice
role: admin
---                    # start of document 2 (also ends document 1)
name: bob
role: viewer
...                    # explicit end of document 2

When --- is required

The document start marker is mandatory in two cases:

  • The document begins with a YAML directive (%YAML or %TAG) — the --- terminates the directive block and begins the document content.
  • You are writing a multi-document stream — every document after the first must start with ---.

For a single-document file with no directives, --- is optional. Many style guides include it anyway as a visual signal that "this is YAML."

Directives — %YAML and %TAG

Directives appear before the --- marker and configure the parser for that document. There are exactly two defined directives.

%YAML

Declares the YAML version the document conforms to. A parser may use this to select which implicit typing rules to apply:

%YAML 1.2
---
strict_booleans: true   # only true/false, not yes/no
yaml.v3 and %YAML

yaml.v3 accepts the %YAML directive but does not change its behaviour based on the version declared — it always applies its own hybrid 1.1/1.2 rules regardless. The directive is parsed and then ignored.

%TAG

Defines a tag handle — a shorthand prefix for long tag URIs:

%TAG ! tag:mycompany.com,2024:
---
credential: !vault secret/db   # expands to tag:mycompany.com,2024:vault

In practice %TAG is rare in config files. It's more common in YAML generated by tools that use custom tag namespaces.

Multi-document streams

A single file can hold several complete YAML documents, each separated by ---. This is used heavily in Kubernetes manifests and test fixtures, but less often in application config files:

---
env: development
db_url: "postgres://localhost/myapp_dev"
---
env: production
db_url: "postgres://prod-host/myapp"

Each document is completely independent — anchors defined in document 1 are not visible in document 2.

Reading multi-document streams with yaml.v3

yaml.Unmarshal only reads the first document in a stream. To iterate all documents, use a yaml.Decoder:

dec := yaml.NewDecoder(file)
for {
  var doc MyConfig
  err := dec.Decode(&doc)
  if errors.Is(err, io.EOF) {
    break
  }
  if err != nil {
    return err
  }
  // process doc
}

Decode returns io.EOF (not an error) when the stream is exhausted. Any other error is a real parse failure.

yaml.Unmarshal silently drops extra documents

If your config file accidentally contains two documents (a stray --- is enough) and you use yaml.Unmarshal, everything after the first --- boundary is silently ignored. Use yaml.Decoder and count documents if you need to enforce single-document files.

Enforcing single-document config files

For a CLI config you usually want exactly one document. Enforce it explicitly:

dec := yaml.NewDecoder(file)
var cfg Config
if err := dec.Decode(&cfg); err != nil {
  return err
}
// Check there is no second document
var extra yaml.Node
if err := dec.Decode(&extra); !errors.Is(err, io.EOF) {
  return fmt.Errorf("config file must contain exactly one YAML document")
}

Gotchas

Anchors don't cross document boundaries

---
base: &defaults
  timeout: 30
---
server:
  <<: *defaults   # error — anchor "defaults" not defined in this document

... does not reset anchor scope

The document end marker ... closes the document but the anchor table is reset only when the next document starts with ---. This is an edge case you will not hit in practice, but it explains why --- and ... are not symmetric.

Empty documents are valid

---
---   # second document is empty (null root node)

yaml.v3 decodes an empty document as a zero-value of the target type — nil for a pointer, empty struct otherwise. A robust parser should check for this.
