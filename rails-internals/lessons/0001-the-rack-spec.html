The Rack Spec

The contract that makes Rails (and everything else) possible

Lesson 1 of ? ~20 min Foundation

Why start here?

Every Rails request — from the moment a socket is accepted to the moment a response is written back — passes through Rack. Understanding Rack is not optional background knowledge: it is the layer beneath ActionDispatch, beneath middleware, beneath everything. When you can read a Rack app, the rest of Rails becomes derivable rather than magical.

Rack is also small. The entire specification fits on one page. That makes it a perfect foundation: high leverage, low surface area.

What is Rack?

Rack is a minimal interface between Ruby web frameworks and web servers. It answers the question: how does Puma (the server) hand a request to Rails (the framework), and how does Rails hand a response back?

Before Rack (circa 2007), every Ruby framework had to write its own adapter for every web server. Mongrel, WEBrick, CGI — each needed custom code per framework. Rack collapsed this into a single, tiny contract. Now any server that speaks Rack can run any framework that speaks Rack.

Web Server (Puma, Falcon, Unicorn) │ │ calls your app with (env) ▼ ┌──────────────────┐ │ Rack App │ ← any callable Ruby object │ .call(env) │ └──────────────────┘ │ │ returns [status, headers, body] ▼ Web Server writes response to socket

That's the whole spec. A Rack app is any Ruby object that responds to #call, accepts one argument (the env hash), and returns a three-element array.

The contract in full

The app

A valid Rack app must respond to call(env) and return a three-element array:

# The simplest possible Rack app
app = Proc.new do |env|
  [
    200,                                    # 1. HTTP status (Integer)
    { "content-type" => "text/plain" },    # 2. Headers (Hash)
    ["Hello, world!"]                     # 3. Body (responds to #each)
  ]
end
The three rules
  1. Status — an Integer ≥ 100
  2. Headers — a Hash. Keys and values must be Strings. Keys must not include status or http-. Header names are case-insensitive per HTTP/1.1 but Rack normalises them to lowercase.
  3. Body — any object responding to #each, which must yield String chunks. Common: Array, File, custom enumerator.

The env hash

The server populates env before calling your app. It contains everything about the incoming HTTP request. Key entries you'll encounter constantly:

Key Value Example
REQUEST_METHOD HTTP verb (String) "GET"
PATH_INFO URL path "/users/42"
QUERY_STRING Query params (raw) "page=2&q=rack"
SERVER_NAME Host name "example.com"
rack.input Request body IO StringIO object
rack.errors Error stream $stderr
HTTP_* HTTP headers (uppercased, -→_, prefixed with HTTP_) HTTP_ACCEPT, HTTP_AUTHORIZATION
Why HTTP_ prefix?
CGI convention. When Rack was designed, many Ruby servers were CGI-based. The HTTP_ prefix distinguishes HTTP headers from server-level keys like REQUEST_METHOD. Rails unwraps this in ActionDispatch::Request — when you call request.headers["Authorization"], Rails is doing the translation for you.

Running a Rack app directly

The rackup command reads a config.ru file and starts a server. This is exactly what rails server does under the hood — Rails generates a config.ru at the root of every app. 

# config.ru — every Rails app has one, look at yours right now
require_relative "config/environment"

run Rails.application

run is a Rack DSL method that sets the terminal app — the one that handles requests that no middleware intercepts. Rails.application is itself a Rack app (it responds to #call).

Try it: open config.ru in one of your own Rails apps right now and look for the run call.

A standalone Rack app (no Rails)

# config.ru
require "rack"

app = Proc.new do |env|
  path = env["PATH_INFO"]
  body = "You requested: #{path}"
  [200, { "content-type" => "text/plain", "content-length" => body.bytesize.to_s }, [body]]
end

run app

Run this with rackup and it works. That's a complete web app. No ActiveRecord, no ActionController, no views.

Where Rails sits in this picture

Rails is not a Rack app that happens to have middleware. Rails is a Rack middleware stack — a series of Rack apps, each wrapping the next, until the innermost one (the router) reaches your controller.

Puma │ ▼ Rack::Sendfile ← middleware │ ▼ ActionDispatch::Static ← middleware │ ▼ ActionDispatch::Executor ← middleware │ ▼ ... (many more) │ ▼ ActionDispatch::Router ← finds your controller action │ ▼ YourController#action ← your code, finally

You can inspect this stack yourself: 

# In a Rails console
Rails.application.middleware.each { |m| puts m }
Key insight
Every middleware in this stack is itself a Rack app. Each one receives env, optionally mutates it, calls the next app in the chain, and optionally mutates the response on the way back out. This means the middleware stack is just a chain of nested function calls — pure Ruby, no magic.

Explore in your own codebase

  1. Open config.ru in a Rails app you own. Find the run call. What does it pass?
  2. In a Rails console, run Rails.application.middleware.each { |m| puts m }. Count how many middlewares are in the stack before your router.
