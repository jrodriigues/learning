Anchors & Aliases — DRY config files

Define a value once, reference it anywhere. The YAML answer to copy-paste configs.

⏱ ~15 min 🎯 &anchor, *alias, merge key <<, yaml.v3 behaviour

The problem anchors solve

Config files often repeat the same values — database credentials shared across environments, a TLS config used by multiple services, a standard set of log fields. Without anchors, you copy-paste and they drift:

# Without anchors — same timeout repeated three times
http:
  connect_timeout: 5
  read_timeout:    30
grpc:
  connect_timeout: 5
  read_timeout:    30
ws:
  connect_timeout: 5
  read_timeout:    30

Anchors let you name a node once and reference it everywhere else — the parser substitutes the value at load time, before your Go code ever sees it.

Anchors — &name

An anchor is placed before the value it names, using an ampersand followed by an identifier:

defaults:
  connect_timeout: &ct 5
  read_timeout:    &rt 30

The anchor name (ct, rt) is scoped to the document. It can be any non-space string that doesn't contain [, ], {, }, or ,. Conventional names use lowercase and hyphens.

Anchors can be placed on scalars, sequences, or entire mappings:

tls_config: &tls
  cert: cert.pem
  key:  key.pem
  ca:   ca.pem

Aliases — *name

An alias references an anchor, using an asterisk:

http:
  connect_timeout: *ct
  read_timeout:    *rt
grpc:
  connect_timeout: *ct
  read_timeout:    *rt

The alias expands to an exact copy of the anchored node. Change the anchor, all aliases update. The YAML parser handles this — Go never sees anchors or aliases, only the resolved values.

Anchors are resolved before Go sees the document

yaml.v3 resolves anchors and aliases during parsing. Your Go structs and maps receive fully expanded values — there is no Anchor or Alias field in the unmarshalled result unless you unmarshal into a yaml.Node tree and inspect it manually.

The merge key — <<

Aliases on scalar values are useful, but the real power is merging mappings. The special key << (two less-than signs) merges the keys from an aliased mapping into the current mapping:

base_db: &base-db
  host:    localhost
  port:    5432
  ssl:     true

read_replica:
  <<: *base-db    # merge all keys from base_db
  host: replica.internal   # override just this one key

Result — read_replica has all three keys from base_db, with host overridden:

read_replica:
  host: replica.internal
  port: 5432
  ssl:  true

Keys defined explicitly in the mapping take priority over merged keys. The merge supplies defaults; local keys override.

Merging multiple anchors

You can merge from a sequence of anchors to compose from several bases:

timeouts: &timeouts
  connect_timeout: 5
  read_timeout:    30

retry: &retry
  max_retries: 3
  backoff_ms:  200

http_client:
  <<: [*timeouts, *retry]   # merge both
  base_url: "https://api.example.com"

When multiple sources provide the same key, the first listed wins (left-to-right). Explicit local keys always win over all merged sources.

yaml.v3 gotchas

The merge key is not standard YAML 1.2

The << merge key is defined in an older YAML 1.1 proposal, not the YAML 1.2 spec itself. yaml.v3 supports it, but not all parsers do. It's safe to use with Go's yaml.v3, but document the dependency if portability matters.

Aliases are copies, not references

yaml.v3 expands each alias into an independent copy of the value. Mutating a Go struct field that came from an alias does not affect other structs that resolved the same anchor — Go values don't share memory just because they came from the same anchor.

Circular aliases are forbidden

a: &a
  b: *a    # error: anchor "a" value references itself

yaml.v3 will return a parse error. Circular structures are not representable in YAML.

Anchor name scope is the whole document

If you define the same anchor name twice in a document, the second definition shadows the first from that point forward. This is valid YAML but confusing — treat anchor names as document-scoped identifiers and keep them unique.

Practical pattern for Go CLI configs

A common pattern: define a _defaults block at the top (anchored) that is never read by Go — its sole purpose is to be merged elsewhere. The leading underscore signals "not a real config key":

_defaults:
  log: &log-defaults
    level:  info
    format: json
  timeouts: &timeouts
    connect: 5
    read:    30

http_server:
  <<: *timeouts
  port: 8080
  log:
    <<: *log-defaults
    level: debug   # override for this service

grpc_server:
  <<: *timeouts
  port: 9090
yaml.v3 and unknown keys

If your Go struct doesn't have a field for _defaults, yaml.v3 silently ignores it by default. To make unknown keys an error (recommended for production configs), use decoder.KnownFields(true) — but you'll need to exclude anchor-only keys or restructure them out of the top-level mapping.
