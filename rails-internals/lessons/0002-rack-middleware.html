Rack Middleware

How Rails intercepts, augments, and short-circuits requests

Lesson 2 of ? ~25 min Foundation

Why middleware is the thing to understand

The Rails middleware stack is not a list of optional plugins bolted on to Rails. It is Rails — the auth check, session handling, request logging, parameter parsing, and static file serving all happen in middleware, before your controller is even found.

When you read a stack trace that starts with ActionDispatch::ShowExceptions or see a 422 you didn't write, you are looking at middleware. Once you understand the middleware pattern, you can trace any behaviour in the Rails request lifecycle to its exact source in actionpack/lib/action_dispatch/middleware/.

The middleware pattern

In Lesson 1 you saw that a Rack app is any object responding to #call(env) that returns [status, headers, body]. A middleware is a Rack app that wraps another Rack app. 

class MyMiddleware
  def initialize(app)   # app = the next Rack app in the chain
    @app = app
  end

  def call(env)
    # 1. Do something before the request hits the next app
    env["my.timestamp"] = Time.now

    # 2. Call the next app — get back [status, headers, body]
    status, headers, body = @app.call(env)

    # 3. Do something with the response before returning it
    headers["x-processed-by"] = "MyMiddleware"

    [status, headers, body]
  end
end

The three-step shape — before / delegate / after — is the core pattern. Every middleware in Rails follows it, from the simplest to the most complex.

Puma calls env into the outermost middleware ┌────────────────────────────────────────────────┐ │ MyMiddleware#call(env) │ │ │ │ mutate env ─────────────────────────────┐ │ │ │ │ │ ┌────────────────────────────────────┐ │ │ │ │ next_app.call(env) │◄─┘ │ │ │ (another middleware, or router) │ │ │ └────────────────────────────────────┘ │ │ │ │ │ mutate response ◄───────┘ │ │ │ └───────────────────────────────────────────────┘ │ returns [status, headers, body] to Puma

The three things a middleware can do

1 — Mutate the environment (request)

Add keys, read or rewrite headers, parse the body, set the current user. ActionDispatch::RequestId generates a unique request ID and sets it in env["action_dispatch.request_id"] before calling downstream.

2 — Mutate the response

Inspect or rewrite the status code, add response headers, buffer and transform the body. Rack::Sendfile rewrites the response body to an X-Sendfile header so Nginx can serve the file directly.

3 — Short-circuit (never call @app)

Return a response immediately without calling the next app. Used for things like serving static assets (ActionDispatch::Static) and health check endpoints. If the file exists, return it. If not, pass through. 

def call(env)
  if env["PATH_INFO"] == "/healthz"
    return [200, { "content-type" => "text/plain" }, ["ok"]]
  end
  @app.call(env)   # everything else passes through
end
Key insight
Short-circuiting is why some requests never reach your controller at all. If ActionDispatch::Static finds a file in public/, it returns it and your entire app — router, controller, ActiveRecord — is never touched. This is pure Rack: the chain terminates at whichever middleware returns first.

Rails's middleware stack

Rails assembles its middleware stack at boot time in railties/lib/rails/application/default_middleware_stack.rb. You can inspect the full ordered list at any time: 

# In a Rails console (rails c)
Rails.application.middleware.each { |m| puts m.name }

# Or from the terminal
bin/rails middleware

A typical Rails 7 app shows roughly 20 middlewares. Here are the ones you'll encounter most:

Middleware What it does
Rack::Sendfile Rewrites file responses to X-Sendfile for Nginx/Apache
ActionDispatch::Static Serves files from public/ — short-circuits before routing
ActionDispatch::Executor Wraps each request in the Rails executor (code reloading, async)
ActionDispatch::Session::CookieStore Reads/writes the signed encrypted session cookie
ActionDispatch::Flash Populates flash from session and sweeps it
Rack::MethodOverride Turns _method=PATCH form params into REQUEST_METHOD
ActionDispatch::Routing::RouteSet The router — dispatches to your controller action
Where to find these in Rails source
All ActionDispatch middlewares live in actionpack/lib/action_dispatch/middleware/. Each file is one middleware class. Reading session/cookie_store.rb or flash.rb is now completely tractable — they're just classes with initialize(app) and call(env).

Build a real middleware in your app

Concretely, here is a middleware that logs request duration. This exact shape — measure, delegate, measure again — appears in production observability middlewares like rack-timer and custom Datadog wrappers. 

# app/middleware/request_timer.rb
class RequestTimer
  def initialize(app)
    @app = app
  end

  def call(env)
    start = Process.clock_gettime(Process::CLOCK_MONOTONIC)
    status, headers, body = @app.call(env)
    elapsed = Process.clock_gettime(Process::CLOCK_MONOTONIC) - start

    headers["x-runtime"] = "%.3f" % elapsed
    [status, headers, body]
  end
end

Register it in config/application.rb:

# config/application.rb
module MyApp
  class Application < Rails::Application
    config.middleware.use RequestTimer
  end
end

use appends to the end of the stack (just before the router). If you need a specific position: 

config.middleware.insert_before ActionDispatch::Static, RequestTimer
config.middleware.insert_after  Rack::Sendfile,          RequestTimer
Order matters
Middleware runs outside-in on the request and inside-out on the response. A middleware inserted before ActionDispatch::Session can't read env["rack.session"] — session hasn't been populated yet. Always check where the key you need is set before deciding where to insert.

Try it now

  1. Create app/middleware/request_timer.rb in a Rails app you own and register it with config.middleware.use RequestTimer.
  2. Run bin/rails middleware — confirm RequestTimer appears.
  3. Hit any endpoint with curl -i http://localhost:3000/ and look for the x-runtime header in the response.
  4. Experiment: move it with insert_before and re-run bin/rails middleware to see the position change.

Reading a real Rails middleware

Open actionpack/lib/action_dispatch/middleware/request_id.rb. It's about 40 lines. You should now be able to read it cold:

  • initialize(@app) — stores the next app
  • call(env) — generates or reads a request ID, sets it on env, calls @app.call(env), then sets it on the response headers too
  • The make_request_id helper normalises the incoming X-Request-Id header or generates a UUID

This is the codebase you'll eventually contribute to. Notice it's just Ruby classes — no metaprogramming, no DSL magic. The middleware pattern is the whole abstraction.

Explore in the Rails source

  1. Read action_dispatch/middleware/request_id.rb in full. You now have all the tools to understand every line.
  2. Run bin/rails middleware in an app you own. Pick two middlewares from the list and find their source files in actionpack/lib/action_dispatch/middleware/.
  3. Look at railties/lib/rails/application/default_middleware_stack.rb — this is where Rails decides which middlewares to include by default and in what order. Notice how it uses conditionals based on config settings to include or exclude things like ActionDispatch::Session::CookieStore.
