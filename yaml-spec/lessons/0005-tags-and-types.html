Tags & Explicit Types

Override implicit typing. Make YAML say exactly what you mean.

⏱ ~15 min 🎯 !!str, !!int, !!bool, !!map, !!seq, custom tags, yaml.v3 tag dispatch

Why tags exist

In Lesson 1 you saw that YAML resolves types implicitly: the unquoted scalar true becomes a boolean, 42 becomes an integer, and so on. Tags are YAML's escape hatch — they let you assert the type of any node explicitly, overriding implicit resolution entirely.

You've already used one implicit tag every time you quoted a value. "42" works because the double-quote style carries an implicit !!str tag. Explicit tags just make this visible:

port:    !!str 5432   # integer literal, forced to string "5432"
enabled: !!str true   # boolean literal, forced to string "true"
ratio:   !!int "8"    # quoted string, forced to integer 8

Tag syntax

A tag is written immediately before the node it annotates, separated by a space:

!<tag-uri> full URI form — !<tag:yaml.org,2002:str> !!shorthand core schema — !!str, !!int, !!float, !!bool, !!null !local local tag — !MyType (application-defined)

The !! prefix is shorthand for the standard YAML tag namespace tag:yaml.org,2002:. In practice you will see !! tags almost exclusively — the full URI form is only needed when writing a YAML parser or schema tool.

The core schema tags

These are the tags defined by the YAML 1.2 Core Schema. yaml.v3 recognises all of them:

!!str string — any scalar value !!int integer — decimal, hex (0x), octal (0o) !!float float — decimal, scientific, .inf, .nan !!bool boolean — true / false only (YAML 1.2) !!null null — null, ~, or empty value !!seq sequence — a YAML sequence (list) !!map mapping — a YAML mapping (object)
# Forcing types that would otherwise be ambiguous
version:    !!str   1.0      # float→string: "1.0" not 1.0
country:    !!str   NO       # bool→string: "NO" not false
zip_code:   !!str   07030    # octal→string: "07030" not 3608
user_count: !!int   "1_000"  # quoted string→integer 1000
Tags vs quoting for strings

Quoting and !!str are equivalent for scalar strings — both force the value to be a string. Quoting is idiomatic YAML; !!str is explicit and self-documenting when the type ambiguity is the point. Use tags when you want the config file to communicate intent ("this is deliberately a string"), quoting when it's just a syntax convenience.

Tags on collections

Tags can annotate sequences and mappings too, though this is less common in config files:

matrix: !!seq
  - !!seq [1, 0]
  - !!seq [0, 1]

metadata: !!map
  created: 2024-01-15
  author:  jrodr

The explicit !!seq and !!map tags add no information that YAML wouldn't infer anyway — they're useful mainly for clarity in generated YAML or when a custom schema reassigns meaning to a tag.

Custom and local tags

A single ! prefix introduces a local tag — an application-defined type hint. The YAML parser preserves the tag string; your application code decides what it means:

credential: !vault secret/myapp/db-password
duration:   !duration "5m30s"
regexp:     !regexp  "^[a-z]+$"

YAML itself doesn't know what !vault means. Your Go code must handle it via a custom Unmarshaler.

How yaml.v3 handles tags

Reading the tag with yaml.Node

To inspect or act on tags in Go, unmarshal into a yaml.Node tree instead of a concrete struct. Each Node carries a Tag field with the resolved tag string:

var root yaml.Node
yaml.Unmarshal(data, &root)
// root.Content[0] is the document node
// root.Content[0].Tag == "!!map", "!!str", "!vault", etc.

Custom Unmarshaler for local tags

Implement yaml.Unmarshaler on your type to intercept the raw node and inspect its tag:

type VaultRef struct { Path string }

func (v *VaultRef) UnmarshalYAML(value *yaml.Node) error {
  if value.Tag != "!vault" {
    return fmt.Errorf("expected !vault tag, got %s", value.Tag)
  }
  v.Path = value.Value
  return nil
}
yaml.v3 normalises !! tags

When you read a node's Tag field, yaml.v3 returns the expanded form: !!str becomes "!!str", not "tag:yaml.org,2002:str". Local tags (!vault) are returned as-is with the single !. Keep this in mind when comparing tag strings in a custom Unmarshaler.

Ignoring unknown tags

If a local tag appears on a node that yaml.v3 can unmarshal into your struct field type, it does so and ignores the tag. The tag only matters if you write an Unmarshaler that inspects it. This means !vault secret/path unmarshals to the string "secret/path" in a plain string field — the tag is silently discarded.

Tags in practice for Go CLI configs

  • Use quoting rather than !!str for the common cases (version strings, country codes, zip codes). It's cleaner and idiomatic.
  • Use !!str explicitly only when the config file is generated by a tool and you want to be defensive about ambiguous values.
  • Use local tags (!vault, !duration, !regexp) when you want config values to carry type-level semantics that Go code must enforce — and implement a matching Unmarshaler.
  • Avoid !!seq and !!map on block collections — they add noise without meaning in normal config files.
